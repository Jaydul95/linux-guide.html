[ myth ]

Linux is only for hackers and programmers

It is some underground system used by people in dark rooms. Totally irrelevant for regular IT work and has no graphical interface.

[ reality ]

Linux is the infrastructure of the modern world

Google, Netflix, Facebook, Amazon — all Linux. Your router likely runs Linux. Android (3 billion devices) is built on Linux. It has full graphical desktops. This is not exotic, it is the foundation.

[ myth ]

You need to memorise hundreds of commands

It is an endless sea of cryptic commands with no intuition. One typo and you have deleted everything. You will never remember any of it.

Click to reveal the reality

[ reality ]

Ten commands get you through the first month

pwd, ls, cd, cat, grep, cp, mv, rm, man, sudo. That is genuinely it to start. The rest you learn as you need it. The man command literally teaches you everything from inside the terminal.

[ myth ]

Breaking things means starting over completely

One wrong command and the whole system is gone forever. It is fragile, unstable, and nobody will help you when it dies.

Click to reveal the reality

[ reality ]

Practice on a VM — break it, snapshot, restore

Linux servers routinely run for years without rebooting. Set up VirtualBox with snapshots and you can experiment freely. Break something? Restore the snapshot in 30 seconds. The community on Ask Ubuntu and Reddit answers every question.

This is just academic — no real-world use

My company uses managed services and cloud platforms with nice dashboards. Nobody actually types commands into servers anymore.

Role breakdown

What each IT role actually does with Linux

Not abstract theory — what you will spend your actual working day doing in each role, and how Linux is involved in every single one.

Most common Linux entry role

Systems Administrator

The sysadmin owns the infrastructure. You provision servers, keep services running, manage users and permissions, handle backups, apply patches, and troubleshoot failures at 2am. On Linux, this is entirely terminal-driven. You will know the filesystem intimately, manage services with systemctl, read logs with journalctl, and write scripts to automate everything you repeat more than twice.

How Linux is used day to day

Managing user accounts, groups, and sudo privileges across dozens of servers simultaneously using SSH and shell scripts
Monitoring system health with tools like htop, sar, and iostat — identifying bottlenecks before they become outages
Configuring and managing services — Apache, Nginx, PostgreSQL, SSH — all controlled via systemctl and config files in /etc
Writing bash scripts to automate log rotation, disk cleanup, and scheduled backups with cron
Applying security patches, configuring firewalls with iptables or ufw, and hardening server configurations
Diagnosing failures by reading system logs in /var/log, tracing process errors, and reviewing kernel messages

$65K – $125K
Senior level with RHCSA/LFCS: $100K–$140K

DevOps Engineer / Site Reliability Engineer

DevOps sits at the intersection of development and operations. You build the pipelines that take code from a developer's laptop to production automatically, safely, and reliably. Every CI/CD system, container, and deployment script runs on Linux. An SRE does the same but with a focus on system reliability, defining SLOs, writing runbooks, and building systems that heal themselves when things go wrong.

How Linux is used day to day

Building Docker containers that run on Linux — writing Dockerfiles, building images, managing registries, debugging container failures
Managing Kubernetes clusters — every node is a Linux machine, every pod runs inside a Linux container, every log is a Linux process output
Writing Ansible playbooks and Terraform configs to provision and configure Linux infrastructure across hundreds of servers at once
Configuring CI/CD pipelines in GitHub Actions, Jenkins, or GitLab CI — all running on Linux build agents
Debugging production incidents by SSH-ing into servers, reading logs, tracing network calls, and analysing memory dumps
Configuring monitoring and alerting with Prometheus, Grafana, and ELK Stack — all Linux-native tooling

Cloud Engineer

Cloud engineers design, build, and manage infrastructure on AWS, Azure, or GCP. The cloud is fundamentally Linux servers with API wrappers. Understanding the OS underneath the cloud platform transforms you from someone who clicks dashboards into someone who understands what is actually happening.

Connecting to cloud VMs via SSH and managing them just like any Linux server — installing software, configuring services, managing users
Writing cloud-init scripts and user data that run on Linux boot to configure new instances automatically at launch
Configuring Linux-based networking — VPC routing, security groups, firewall rules — which map directly to iptables and kernel networking
Deploying and managing containerised workloads on ECS, GKE, or AKS — all Linux containers under the hood
Troubleshooting cloud resource issues that always trace back to Linux-level problems — disk I/O, memory pressure, process limits, network latency
Managing AWS Lambda and serverless runtimes, which run on Amazon Linux containers behind the scenes

Cybersecurity Analyst / Penetration Tester

Cybersecurity is built on Linux. The tools, the workflows, the certifications — all assume Linux fluency. Networking meets Linux

Network Engineer

Network engineers manage the infrastructure that connects systems — routers, switches, firewalls, load balancers, VPNs. Increasingly, these devices run Linux or Linux-derived operating systems. Cisco IOS has Linux under the hood. Most modern firewalls are Linux-based. SD-WAN platforms are Linux. Understanding Linux networking at a low level — the kernel's networking stack, routing tables, and iptables — makes you a significantly better network engineer.

How Linux is used day to day

Using Linux networking tools to diagnose connectivity issues — ip, ss, traceroute, tcpdump, nmap — the same tools used on both Linux servers and many network devices
Managing Linux-based firewalls — pfSense, OPNsense, and UFW are all Linux under the hood with iptables doing the actual filtering
Configuring VPN tunnels (OpenVPN, WireGuard) which run as Linux daemons and require Linux administration knowledge to operate
Using Python scripts running on Linux to automate network device configuration via SSH and NETCONF protocols
Setting up network monitoring with tools like Nagios, Zabbix, or Prometheus — all running on Linux servers

Production runs on Linux

Most software developers work on macOS or Windows but deploy their code to Linux servers. The developers who can do this — who understand the environment their code runs in — are considerably more effective and are treated as senior contributors regardless of their title.

Using WSL2 or a Linux VM as the local development environment — running databases, message queues, and local Kubernetes clusters that mirror production

Writing Dockerfiles that build Linux-based container images — understanding the base layer, layer caching, and runtime behaviour

Choosing your distribution

Which Linux should you start with?

Linux comes in many versions called distributions or distros. They all use the same Linux kernel but differ in tooling, package management, and intended use. Pick one and commit to it — switching later is painless once you know the fundamentals.

Ubuntu
Best for: Beginners, developers, servers
The most popular Linux distribution by a wide margin. Huge community, excellent documentation, and supported by virtually every cloud platform and hosting provider. Ubuntu is the default choice for learning and is used extensively in production.
Package manager: apt  |  Based on: Debian
Debian
Best for: Stability-focused servers
The grandfather of many distros including Ubuntu. Prioritises rock-solid stability over cutting-edge software versions. Servers running Debian are famous for uptimes measured in years. Learning Debian skills directly transfers to Ubuntu and vice versa.
Package manager: apt  |  Based on: Debian
Red Hat / RHEL
Best for: Enterprise corporate environments
Red Hat Enterprise Linux is the standard in large corporate and government environments. Most RHCSA and RHCE certification study uses RHEL or its free clones. If your company is an enterprise, you will likely encounter RHEL or one of its derivatives.
Package manager: dnf  |  Family: Red Hat
AlmaLinux / Rocky Linux
Best for: Free RHEL alternative
Free, community-maintained clones of RHEL that are binary compatible — packages built for RHEL run unchanged. AlmaLinux and Rocky Linux emerged when CentOS was discontinued and are now the default free RHEL alternatives for learning and production use.
Package manager: dnf  |  Based on: RHEL source
Kali Linux
Best for: Cybersecurity, ethical hacking
Purpose-built for penetration testing and security research. Ships with 600+ pre-installed security tools including Metasploit, Wireshark, Nmap, Burp Suite, and hashcat. Not recommended as a daily driver for beginners — use it specifically for security practice after learning Linux basics.
Package manager: apt  |  Based on: Debian
Arch Linux
Best for: Advanced users who want full control
A minimalist distro where you build everything from scratch. No installer that makes decisions for you — you partition the disk, install the bootloader, configure the network, and choose every package. Technically demanding but produces deeply knowledgeable Linux users. Not for beginners.
Package manager: pacman  |  Based on: Independent

Recommendation: Start with Ubuntu 22.04 LTS or 24.04 LTS. Install it in WSL2 on Windows or as a VirtualBox VM. It has the largest community, the best beginner documentation, and is used in most online tutorials. Once you are comfortable, try AlmaLinux or Rocky Linux to learn the Red Hat ecosystem before pursuing RHCSA certification.

Learning path

Eight steps from zero to confident

Each step builds directly on the one before it. Do not skip ahead — the gaps will show up later. Click any step to expand it and see the specific commands to learn and why they matter.

01
Week 1–2  ·  Foundation

Set up your Linux environment

Before anything else, you need a Linux environment you can freely experiment in. The best option for Windows users is WSL2 — Windows Subsystem for Linux. Open PowerShell as administrator and run wsl --install, then restart. Ubuntu installs automatically. Mac users already have a Unix terminal — open Terminal and you are 80% there. Alternatively, download VirtualBox for free and install an Ubuntu ISO as a VM. This is the safest learning environment because you can take snapshots and restore them when things go wrong.

wsl --installInstall WSL2 with Ubuntu on Windows (run in PowerShell as Admin)
wsl -l -vList installed Linux distributions and their WSL version
uname -aDisplay Linux kernel version and system information
whoamiPrint the current logged-in username
sudo apt updateRefresh the package list (first command to run on a fresh install)
Tip: Set up VirtualBox with a snapshot immediately after installing Ubuntu and before doing anything else. Name it "Clean install." You now have a restore point for any experiment that goes wrong.

02
Week 2–4  ·  Navigation

Master terminal navigation and basic commands

The terminal is the core skill. Every other thing you learn happens in here. Start by getting comfortable navigating the filesystem. Understand that in Linux, your current location (called the working directory) matters for every command you type. Learn to move around, list contents, and examine files. The man command is your teacher — type man followed by any command name and get the full manual for that command, right inside the terminal.

pwdPrint working directory — shows where you currently are
ls -laList all files including hidden ones, with permissions and sizes
cd /path/nameChange to a specific directory; cd ~ goes home; cd .. goes up one level
cat filenameDisplay the full contents of a file in the terminal

grep "word" fileSearch for a pattern inside a file — one of the most used commands ever
man lsRead the full manual for any command — replace ls with any command name
historyShow previously typed commands; use up arrow to cycle through them
Tip: Press Tab to autocomplete file and directory names. Press Ctrl+R to search your command history. These two shortcuts alone will save you hundreds of keystrokes every day.

03
Week 3–5  ·  File system

Understand the Linux file system hierarchy

Linux organises everything in a single directory tree that starts at / (pronounced "root"). Unlike Windows, which has separate drive letters (C:, D:), everything in Linux is under this one tree — including physical drives, USB sticks, network shares, and even hardware devices. Knowing what lives where is essential for configuration, troubleshooting, and system administration. The phrase "everything is a file" in Linux is literally true — even hardware devices like hard drives (/dev/sda) are represented as files.

/etcSystem-wide configuration files — every service stores its config here
/home/usernamePersonal files for each user — equivalent to C:\Users\Name on Windows
/var/logSystem and application logs — first place to look when something breaks
/tmpTemporary files — automatically cleared on reboot
/procVirtual filesystem showing live kernel and process information
/devDevice files — /dev/sda is your first hard disk, /dev/null discards everything
df -hShow disk space usage for all mounted filesystems in human-readable format
find / -name "*.log"Search for all files with .log extension starting from root directory
Tip: Read the log files in /var/log regularly, even when nothing is broken. Understanding what normal looks like makes it much easier to spot what abnormal looks like when an incident occurs.

04
Week 4–6  ·  Security model

Users, groups, and permissions

Linux was designed from the ground up as a multi-user system with security built into every file and directory. Every single file and directory has three permission groups — the owner (u), the group (g), and everyone else (o) — and each group can have read (r), write (w), and execute (x) permissions. This system is how Linux enforces security, and understanding it is fundamental to both administration and cybersecurity. The root user is the superuser with unrestricted access to everything — do not run as root unless absolutely necessary.

ls -l fileShow permissions — e.g. -rwxr-xr-- means owner can rwx, group can rx, others r only
chmod 755 fileSet permissions numerically: 7=rwx, 5=rx, 4=r — 755 is common for executables
chmod +x script.shAdd execute permission to a file — makes it runnable as a program
chown user:group fileChange the owner and group of a file
sudo commandRun a single command as root — the safe way to do admin tasks
useradd -m usernameCreate a new user with a home directory (-m flag)
passwd usernameSet or change a user's password
groups usernameShow which groups a user belongs to
Tip: Never run as root by default. The habit of using sudo only when needed is what separates cautious administrators from people who accidentally delete /etc one day. Use sudo, do the specific thing, then go back to being a normal user.
chmodchownsudouseraddpasswdgroups
05
Week 5–7  ·  Software management

Package management and software installation

Linux uses package managers to install, update, and remove software safely and cleanly, with no hunting for .exe files or fighting with installers. The package manager knows about dependencies — software that other software needs — and handles everything automatically. On Ubuntu and Debian, the tool is apt. On Red Hat, RHEL, AlmaLinux, and Rocky Linux, it is dnf. Snap and Flatpak are newer universal package formats that work across distributions.

sudo apt updateRefresh the list of available packages and their versions
sudo apt upgradeInstall all available updates for installed packages
sudo apt install nginxInstall the nginx web server (replace nginx with any package name)
sudo apt remove nginxUninstall a package, keeping its configuration files
apt search pythonSearch for packages by name or description

Week 6–9  ·  System management

Processes, services, and system monitoring

Every program running on Linux is a process with a unique process ID (PID). As an administrator you need to see what is running, how much CPU and memory it is consuming, and be able to start, stop, or kill processes as needed. Modern Linux systems use systemd to manage long-running services — the daemon processes that start on boot and keep running in the background, like web servers, databases, and SSH. Understanding systemctl is essential for any server administration work.

topInteractive real-time view of running processes and resource usage — press q to quit
htopBetter version of top with colours and easier controls — install with apt install htop
ps auxSnapshot list of all running processes with their PID, CPU, and memory usage
kill 1234Terminate process with PID 1234 (use ps or top to find the PID first)
kill -9 1234Force-kill a process that is not responding to the normal kill signal
systemctl status nginxCheck if the nginx service is running and see recent log output
systemctl start nginxStart a service (also: stop, restart, reload)
systemctl enable nginxMake a service start automatically on every boot
journalctl -u nginxView all log output for the nginx service — add -f to follow live
free -hDisplay available and used memory in human-readable format
Tip: When a service fails, the first command to run is always journalctl -u servicename --since "10 minutes ago". This shows you what the service was doing right before it failed, which is where the actual error message lives.

Week 8–12  ·  Networking

Networking fundamentals from the command line

Linux provides a complete suite of networking tools that are used professionally to diagnose connectivity, inspect traffic, manage firewall rules, and connect to remote systems. These tools map directly to real networking concepts — IP addresses, routing, ports, protocols — making Linux networking practice one of the best ways to understand how networks actually work, not just in theory. SSH is the universal tool for connecting to remote Linux servers and is used by every IT professional who manages infrastructure.

ip addr showDisplay all network interfaces and their IP addresses (replaces ifconfig)
ip route showShow the routing table — where network traffic is being directed
ping -c 4 google.comTest connectivity to a host — sends 4 ICMP packets and reports response times
traceroute google.comTrace the network path and every hop between you and a destination
ss -tulnShow all listening ports and their services (TCP/UDP listening, numeric output)
nmap -sV 192.168.1.1Scan a host's open ports and identify which service is running on each
curl -I https://site.comMake an HTTP request and show only the response headers — great for testing
ssh user@serverConnect securely to a remote server — the fundamental tool of server administration
ssh-keygen -t ed25519Generate a cryptographic key pair for passwordless SSH authentication
scp file.txt user@host:~Securely copy a file to a remote server over SSH
Tip: Set up SSH key-based authentication instead of passwords immediately. Run ssh-keygen, then ssh-copy-id user@server. You will never type a password to SSH again, and your connection will be more secure than any password login.

Week 10–16  ·  Automation

Shell scripting — automate everything

Shell scripting is where Linux fluency becomes a career superpower. If you do something manually more than twice, you should write a script. Scripts can run automatically on a schedule using cron, respond to system events, process log files, back up data, check server health, and deploy applications. Bash scripting is not programming in a traditional sense — it is the art of combining the Linux commands you already know into automated workflows. The learning curve is gentle and the career return is extraordinary.

#!/bin/bashThe shebang line — always the first line of a bash script, tells Linux which interpreter to use
chmod +x script.shMake the script executable before running it
./script.shRun a script from the current directory
name="Linux"Declare a variable — no spaces around the equals sign
echo "Hello $name"Print text — use double quotes to allow variable expansion
if [ condition ]Conditional logic — check if a file exists, a number is greater, a string matches
for f in *.logLoop over all .log files in the current directory
crontab -eEdit your cron schedule — automate scripts to run at any time interval
command | tee log.txtRun a command and save its output to a file while also displaying it in the terminal
Tip: Your first real script project should solve an actual problem you have. Back up a directory you care about, check disk space and email you when it gets low, or rename 500 files according to a pattern. Solving a real problem is 10 times more memorable than running through exercises.
